# INV-1 波纹衰减 (Ripple Decay)

**分类**: 已确认 invariant
**Target skill**: `harness-dev`

## 模式定义

一个改动从源头发出时是"强波纹"——开发者清楚它会影响 A/B/C 三个下游。随着改动沿调用链向外传播，agent 对"这里还要不要改"的意识逐层衰减；传到第四层时，改动已被遗忘，该层保留着旧语义。

核心反差：**agent 改动源头时很清醒，传播时很健忘**。

## 典型形状

- 改了某个字段的序列化格式，改了直接调用方，但漏了通过消息队列异步消费的下游。
- 改了 API 请求 schema，改了 handler，但漏了 OpenAPI 文档 / 前端类型定义 / 集成测试 fixture。
- 改了某个默认值，但配置迁移脚本里还写着旧值作为"兜底"。

## 检测信号

- 改动 commit 只触及 2-3 个文件，但函数/字段被 10+ 处引用
- 某层下游的测试在改动后仍 pass（可疑——应该要么被更新要么至少失败）
- 修完一个 bug 后几天又出现"新"的同形状 bug

## 缓解动作

**注入到 `harness-dev`**：

1. 任何契约类改动（API / schema / event / 配置项）必须先跑**全量消费方 grep**，列出 N 个位置，对每个位置显式打 "改 / 不改 + 原因" 标签。
2. "不改 + 原因"的位置数 > 0 时，原因必须写在 PR 描述里，供审查核验。
3. Agent 在 diff 只涉及 1-2 个文件时要警惕："是真的就这 2 处，还是我漏看了消费方？"

## 与 INV-4 的关系

INV-4（真相源分裂）是"同一事实被写在多处"的**存量**问题；INV-1 是"一次改动未传播到全部消费方"的**流量**问题。实际上 INV-4 很多时候是 INV-1 积累出来的——每次都漏一个下游，下游就变成新的事实副本。
