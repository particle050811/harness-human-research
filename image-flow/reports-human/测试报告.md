# 人工测试报告

## 测试环境

**被测模型选用 deepseek-v4-pro**，理由如下：

- 运行稳定且价格相对低廉。若使用廉价中转的 GPT 或 Claude，中途断流、重启很麻烦；若用正价 API，价格负担不起。
- Claude Fable 5 相比 deepseek-v4-pro 明显更强，因此可以用 Claude Fable 5 来分析 deepseek-v4-pro 的表现。

**Claude Code 固定 2.1.152 版本**：后续版本更新改了参数，会导致 deepseek-v4-pro 无法正常调用 subagent。

**环境隔离**：用脚本复制独立文件夹与 skills，供被测会话开发使用。

## 测试过程遇到的问题

### 问题一：skill 装而不用

给 deepseek 提供了 superpowers，但在 200+ 轮编码中 0 次调用。

**解决方法**：主动指定每轮需要调用的 skill。

### 问题二：一次性实现导致需求遗漏

deepseek-v4-pro 试图一次性实现所有代码，且写出的计划包含大量代码，导致爆 1M 上下文、遗漏需求。使用 superpowers 的效果甚至不如不用。

**解决方法**：让 Claude Fable 5 逐轮发送剧本提示词，每轮结束后读模型总结、实地核验、对照 spec 验收，再附**大方向指导**发下一轮（有可用 skill 时会指定要调用的 skill）。

### 问题三：被测模型越界写入评测仓库

被测模型会把上级目录猜成项目根，将产物写进评测仓库本身（嵌套 git 仓库挡不住直接猜绝对路径）。

**解决方法**：写了 PreToolUse hook（path-guard）：文件类工具只放行 run 目录与 /tmp，Bash 命令出现 run 目录外的评测仓库绝对路径即拒绝，拦截理由经 stderr 回灌给被测模型让其自行改用项目内路径。实测拦截后模型当轮即纠正，后续不再越界。

### 问题四：偶发零产出轮

deepseek-v4-pro 偶发提前收 turn：发出提示词后只读了几个文件就结束，无文本无改动（另有一次空转约 35 分钟无产出），与所用 skill 变体无关。

**解决方法**：监工每轮验收时核对本轮是否有实际产出（输出 + 文件改动），发现零产出即以"上轮无产出"提醒重发，重发后均正常完成。

### 问题五："四检全绿"不等于可用

compile/check-types/lint/test 四项检查全部通过的产物，仍可能存在侧栏白屏等配置缝隙类缺陷——这类问题只有运行时能暴露。

**解决方法**：评分前固定跑 `eval-test.sh` 拉起 VS Code 扩展开发宿主，做 F5 侧栏运行时检查，不以静态检查结果作为可用性依据。

### 问题六："UI 文案逐字符合"是模型稳定盲区

spec 里规定了逐字格式的界面文案条款（如进行中卡片标题），各 skill 变体的首批 run 全部中招，且模型的轮内自验和终轮审查都发现不了，只能靠监工逐字对照才暴露——该盲区属于 deepseek-v4-pro 本身，与变体无关。（例外：gstack 重跑中该条款首次自发做对，单 run 样本暂无法区分是工具链生效还是模型随机性。）

**解决方法**：监工验收时对此类条款逐字对照并反馈返工；并计划在剧本的验证轮文本里统一加一句"界面文案与 spec 给出的格式逐字对照"，把核对前移到模型自查。

### 问题七：监工验收尺度不一致影响跨 run 可比性

不同 run 的监工逐轮验收若无固定清单、只凭当轮里程碑主线抽查，会导致部分红线缺陷漏报到报告期，且"监工反馈次数"指标在 run 间不可比（差异来自监工而非模型）。

**解决方法**：把 spec 红线条目沉淀为固定核对清单，每轮验收按"当轮里程碑条款 + 已实现红线全量回归"两段执行。

gstack 重跑已验证该方法有效：3 处红线违例全部在对应里程碑的下一轮获得反馈、当轮修复并经回归保持，最终红线零遗留；"监工缺漏反馈次数"恢复跨 run 可比口径。

## 各 skill 变体性价比对比（监工模式，round3）

四个 run 统一条件：同一 spec 全量 M1~M8、deepseek-v4-pro、Claude Code 2.1.152、监工模式逐轮驱动、剧本逐轮对应。empty（无 skill）为基线。

| 指标 | empty（基线） | superpowers | openspec | gstack（重跑*） |
|---|---|---|---|---|
| 未缓存输入 token | 148.7 万 | 369.7 万（2.49×，且 35 分钟废轮未计入） | 257.8 万（1.73×） | 591 万（3.97×） |
| 输出 token | 58.0 万 | 48.9 万 | 62.0 万 | 74 万 |
| 输入总处理量（含缓存读） | 4738 万 | 5424 万（1.14×） | 7031 万（1.48×） | 7870 万（1.66×） |
| 各轮活跃时长 | 51.5 分钟 | 67.2 分钟（另有 35 分钟废轮） | 60.4 分钟 | 54.6 分钟 |
| 红线遗留违例（评测员读码核对） | 1 处不符合 | 1 处不符合 + 2 处偏差 | **0** | **0** |
| 模型自发 git 提交 | 0 | 9 次 | 0 | 10 次 |
| 工程结构 | 11 文件，最大单文件 658 行，lint 不覆盖前端 | 19 文件，最大 317 行，检查全覆盖 | 16 文件，最大 448 行，检查全覆盖 | 19 文件，最大 363 行，检查全覆盖 |

\* gstack 列采用重跑数据（round3-05）：首跑（round3-04）因监工尺度不一致 + 工具链降级两个混淆因素作废。重跑落实了 HOME 重定向与红线清单后执行；其前两轮含评测员侧误操作（重复 prompt、误杀进程后续推），未缓存输入与时长指标略偏高。

**结论**：

- **openspec 性价比最高**：是唯一红线零遗留的变体，代价是输入总处理量为基线的 1.48 倍（工件常驻磁盘、每轮重读换来上下文压缩免疫——这正是它对症"长尾需求丢失"根因的机制）。如果在意功能正确性，多花的这部分 token 是值的。
- **superpowers 性价比存疑**：未缓存输入是基线的 2.49 倍（最贵的部分），换来的是工程过程质量（逐里程碑提交、模块拆分、检查全覆盖），而在功能正确性上与 empty 打平（各 1 处违例）。大头开销来自每轮重新注入 skill 正文 + 回读 3576 行的巨型计划文档。
- **empty 基线本身不差**：监工模式逐轮驱动 + 验收反馈已经兜住了大部分质量，skill 的净增量比预想小——这是本轮评测最重要的发现之一：**驱动方式的影响大于 skill 本身**（对照问题二：自主模式下用 superpowers 甚至不如不用）。
- **gstack 重跑后功能正确性追平 openspec，但开销最高**：红线零遗留（与 openspec 同为各 3 次监工反馈后达成），且两条历史性 UI 盲区条款首次自发做对、自发逐里程碑提交（10 次中文提交信息）。代价是未缓存输入为基线的 3.97×（四变体最高，主因是每轮 `/spec`、`/review` preamble 的数十条工具探测 + 巨型 skill 正文），总输入处理 1.66×。另外 HOME 重定向后其 bin 工具链虽全程可解析（188 次调用 0 失败），但 `~/.gstack` 状态层（checkpoint/learnings）始终未启用——gstack 的核心卖点仍未真正参与，按当前形态其性价比低于 openspec。

注意：以上对比每个变体只有 1 个 run 样本，且 F5 运行时验收均未补做，结论方向可信、定量数字不宜过度解读。
