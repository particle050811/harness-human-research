## 测试背景

开发一个名为 **image-flow** 的 VS Code 扩展：
用户在 Markdown 里写提示词，通过侧栏触发调用 AI 绘图API生成图片，图片自动下载，并在侧栏中浏览任务进度、历史结果与素材库。


**被测模型选用 deepseek-v4-pro**，理由如下：

- 运行稳定且价格相对低廉。若使用廉价中转的 GPT 或 Claude，中途断流、重启很麻烦；若用正价 API，价格负担不起。
- Claude Fable 5 相比 deepseek-v4-pro 明显更强，因此可以用 Claude Fable 5 来分析 deepseek-v4-pro 的表现。

**Claude Code 固定 2.1.152 版本**：后续版本更新改了参数，会导致 deepseek-v4-pro 无法正常调用 subagent。

**环境隔离**：用脚本复制独立文件夹与 skills，供被测会话开发使用。

## 测试过程遇到的问题

### 问题一：skill 装而不用

（round1 中）给 deepseek 配了 superpowers 这个 skill，但在 200 多轮编码里一次都没用过。

**解决方法**：主动指定每轮需要调用的 skill。

### 问题二：一次性实现导致需求遗漏

（round2 中）deepseek-v4-pro 试图一次性实现所有代码，且写出的计划里塞了大量代码，导致遗漏需求。使用 superpowers 的效果甚至不如不用。

**解决方法**：让 Claude Fable 5（模拟真实用户逐轮验收）逐轮发送剧本提示词，每轮结束后读模型总结、实地核验、对照需求文档验收，再附**大方向指导**发下一轮（有可用 skill 时会指定要调用的 skill）。

### 问题三：被测模型越界写入评测仓库

（round3-01 中）被测模型会把上级目录猜成项目根，将产物写进评测仓库本身

**解决方法**：写了一个hook 拦截非法路径

## Bug数量一行的额外说明

- gstack和wow-harness 有重大bug，无法启动。额外跑了一轮，让deepseek-v4-pro 自己修复
- wow-harness 修复启动问题以后 还有 点击api key 链接插件崩溃的重大bug
- superpowers 图片加载有问题
- openspec 缺少添加素材库的功能
- base 列表不设最大高度，生成按钮被顶到最下方

## 结论
挂上 skills 后，代码组织明显更规范（文件拆分更细、检查覆盖更全），但最终质量没有明显提升。
没用skills的base只有界面有问题，而其余用了skills的都有影响使用的重大bug，甚至可以说是倒退。
而代价是输入、输出 token 量大幅上涨。

## 猜想与改进

skills 表现不如 base，原因可能有以下几点：

- 评测一开始就提供了完整的需求文档，`/brainstorm` 这类用于厘清需求的 skill 失去了发挥空间。
- 所有编程操作都在同一个对话框中完成，而 skills 会大幅撑高上下文，更频繁地触发自动 compact，反而丢失了关键信息。
- 部分 skill 内容冗余，白白占用了宝贵的上下文注意力。
- AI 往往能一气呵成写完代码而不报编译错误，人类却做不到；某些 skill 照搬人类的编码流程，对 AI 而言其实多此一举。
- harness 是与模型协同进化的——Claude Code 经过了模型的后训练打磨，而第三方 skills 没有。

更深一层看，harness 的作用在于补足 AI 本身的短板。受限于有限的上下文，AI 最欠缺的是对项目的全局把控。因此需要人来替它整理约束、维护记忆：把最有用的部分沉淀进 CLAUDE.md，并不时用 `/requesting-code-review`、`/tech-debt-audit` 梳理项目的整体架构。

由此得出两条改进方向：

- 把完整任务拆分为多个功能模块，分散到多个对话框中分别执行。
- 尽量克制使用 skill，只在关键环节才启用。
