口径保留：superpowers 首次尝试（约 35 分钟）的会话文件已删，其消耗**未计入**，为不完全统计；openspec 的 M5 零产出轮已计入；gstack 重跑的计划轮含 2 条误重发 prompt、M1 轮含 2 次误杀续推，数字偏高；wow-harness 含剧本外 R12 补充轮（输出 2.2 万 / 新输入 41.2 万），且其常驻 hooks 每次工具调用注入提示，输入开销形态与按需 skill 不同。

分布特征（细节见各 run 报告主题三）：

1. **计划轮开销悬殊**：superpowers 的 /writing-plans 轮输出 16.6 万 token（占其全程输出 34%）、产出 3576 行计划文档；empty 计划轮仅 1.5 万 token。
2. **执行轮未缓存输入**：superpowers 各执行轮稳定 30.9 万~45.9 万（每轮重注入 skill 正文 + 回读巨型计划文档）；empty 多数轮低于 10 万。
3. **openspec 以重读换常驻**：每个 `/opsx:apply` 轮先跑 CLI 再重读 proposal/design/delta-spec/tasks 等工件，输入总处理量高但工件结构稳定、缓存命中好，未缓存输入比 superpowers 低（1.73× vs 2.49×）。
4. **输出端有计划者更省**：superpowers 扣除计划轮后执行期输出 32.2 万，低于 empty 的 56.5 万——有计划在手试错与重写少，但省下的输出远不抵多花的输入。

**流程开销换来了什么**（superpowers vs empty 的代表性结论）：以未缓存输入 +221 万、活跃时长 +30% 为代价，可量化收益是逐里程碑提交的完整版本史（9 vs 0）、更细的模块拆分（19 vs 11 文件）、lint/类型检查全覆盖、监工反馈少 1 次；而 spec 覆盖正确性维度两边打平。监工模式下 skill 的净收益主要体现在**工程过程质量**，而非功能正确性。openspec 是这一结论的有力补充：其 spec 常驻工件机制换来了红线零遗留（对症 round2-01 的"执行期离场"根因），代价是三者中最高的输入总处理量。

## 主题四：过程行为与 skill 触发对比

- **skill 触发形态**：superpowers 每轮带 `/skill名` 前缀 + SessionStart hook；openspec 的 `/opsx:*` 内嵌 CLI 调用、每轮实际执行；gstack 仅首尾轮（`/spec`、`/review`）有 skill 参与，中间八轮无对应 skill；wow-harness 是常驻治理层（16 个项目级 hook 全程生效 + Gate 状态机），/lead、/harness-dev 前缀只是显式入口，收尾汇报被治理模板约束为固定三段结构。对照 round1 自主模式"skill 可见但零调用"，**监工模式 + 命令前缀是 deepseek-v4-pro 实际用上 skill 的前提**。
- **git 提交行为**：自发提交 0 / 9 / 0 / 10 / 11 次。提交习惯完全由 skill 流程驱动（executing-plans 每任务要求 commit、gstack/wow-harness 流程同类）；openspec 的进度载体是 `tasks.md` checkbox 而非版本史，与 empty 同为零提交。注意 wow-harness 以 git 门禁为核心机制，但其 Stop hook 实测强度有限（untracked 不计、阻塞 1 次即被 TTL 闸放行），git 卫生类监工反馈反而是 round3 各 run 中最多的 3 次（见 round3-06 主题三）。
- **终轮自审（/review 或同构无前缀轮）**：empty 自查 6 项修 5、superpowers 3 项全修、openspec 8 项含 2 处真缺陷、gstack 首跑抓出文件覆盖 CRITICAL、gstack 重跑抓出 2 处真缺陷 + lint 清零、wow-harness 抓出 P1 重复定义 + 4 项 P2 行为对齐。共性：**所有 run 的终轮自审都没召回各自遗留的 UI 逐字类违例**，该档缺陷只有监工逐条对照 spec 才暴露。另注意 gstack 重跑的审查轮没有红线级"剩余猎物"，是红线清单使监工验收前移的预期副作用，不宜跨 run 直接比较 /review 捕获量。
- **DeepSeek 偶发零产出/无响应**：superpowers 首次尝试 35 分钟无产出、openspec M5 轮 1.8 分钟即空收 turn、wow-harness M6 轮发题后 142.7 分钟零响应后自行恢复，均与变体无关，系后端偶发；重发或等待后均正常。
- **上下文压缩的对冲**：五个 run 各发生 1~2 次压缩且均能续完，监工逐轮反馈持续把 spec 出处送回上下文，缓解了 round2 固定剧本下"压缩后只剩计划切片"的失忆问题。

## 主题六：可比性边界（引用上述对比时必须带的保留）

1. **HOME 重定向仅 gstack 重跑启用**：empty/superpowers/openspec 均无 `$RUN_DIR/.home` 重定向，环境不完全对等；对其余变体无已知影响（它们不依赖 `~` 路径），但严格对照须同条件复测。
2. **红线回归清单自 gstack 重跑才生效**：此前各 run 的监工核对覆盖面依赖当轮抽查，"遗留违例数"与"监工缺漏反馈次数"两指标在清单前后口径不一（gstack 首跑的 2 处遗留部分归因监工，见 round3-04 主题四）。
3. **superpowers token 为不完全统计**：首次尝试会话文件已删，约 35 分钟消耗缺失。
4. **path-guard 上线时点不同**：empty 早于上线、superpowers 自第 2 轮起、openspec/gstack 全程生效；hook 对缓存命中的影响未单独验证（round3-02 改进二）。
5. **gstack 重跑前两轮指标失真**：评测员通知误判与时钟回跳导致计划轮 prompt 污染、M1 轮两次误杀续推（round3-05 主题四），该两轮时长/压缩/token 不可与其它 run 直接比较。
6. **运行时验收缺位**：empty/superpowers/openspec/gstack 四个 run 未跑 `eval-test.sh`，红线表中"防闪烁"等运行时表现仅有读码结论，"四检全绿"在 empty（lint 不覆盖前端）与其余 run 之间含金量不同。
7. **wow-harness 轮次与红线核对口径不同**：其红线读码核对由评测员在 11 轮结束后补做（round3-06 改进二），标签页违例靠剧本外 R12 补充轮修复——"遗留违例 0"与"12 轮"两项指标与其它 run 的轮内核对口径不可直接比较；其"监工缺漏反馈"中 3 次为 git 卫生类，与其它 run 的缺陷类反馈性质不同。另 wow-harness 为 drop-in 档（独立审查 agent 缺失、{{PROJECT_NAME}} 占位未替换），治理层以降级形态受测。
