## 主题四：异常事件

本 run 被测模型侧无异常（path-guard 0 拦截、无轮次中断），异常集中在评测员侧基础设施：

- **后台任务通知双发/提前**：R1 发出后收到 0 字节输出的假"completed"通知，评测员误判断流，向存活会话重发 2 条 prompt（重复 `/spec`、"继续完成 /spec 计划任务"），被 queue-operation 排队执行——计划轮被污染（内容中性，但促成提交时点提前，时长指标失真）。
- **WSL2 时钟回跳 + jsonl 写缓冲**：M1 轮中评测员用超前约 45 分钟的时钟对比会话 jsonl 的 mtime，误判会话停滞并 kill 进程（两次），以"上一条会话意外中断，继续完成 M1 剩余工作"续推——停滞很可能是假象，M1 轮时长/压缩指标失真。
- **Monitor 工具不可用**：沙箱 PID 命名空间隔离导致 pgrep 看不到被测进程，对 /tmp 信号文件的判断亦出过假阳性，中途弃用。
- 上述事件后评测员改用固定纪律：每条通知到达先 `ps -p <PID>` 校验进程真实状态，进程存活绝不重发；等待统一用前台有界循环（10 秒粒度、10 分钟上限）。R3 起九轮再无误操作。

## 对应改进

### 改进一：评测员后台等待基础设施固化为操作纪律

**问题**：主题四所列三类基础设施缺陷（通知双发、时钟回跳、Monitor 沙箱隔离）在本 run 前两轮造成 1 次 prompt 污染与 1 次误杀进程，污染了计划轮与 M1 轮的时长/压缩指标。

**方案**：把本 run 后段验证有效的纪律写进 CLAUDE.md 监工流程节：round 发起后以 pgrep 取 PID 并前台有界等待进程退出；任何通知到达先校验进程状态再行动；不使用 Monitor；不以文件 mtime 与本机时钟差值判停滞。

**预期效果**：消除评测员侧对被测会话的误干预，时长/压缩指标全轮可信。

### 改进二：gstack 状态层未启用，工具链生效程度需取证收口

**问题**：主题三所述，HOME 重定向后 bin 工具链可解析可调用，但 `~/.gstack/` 状态目录（sessions/learnings/timeline）全程未产生，gstack 的 checkpoint/learnings 等状态型能力实际仍未参与，"工具链全功能受测"的目标只达成一半，且原因未定位（telemetry 关闭时的 no-op？preamble 分支未走到？）。

**方案**：从会话 jsonl 解析 preamble 复合命令的完整 stdout/stderr，定位状态写入分支未执行的原因；必要时在 setup 中预建 `~/.gstack/` 骨架或显式开启相关配置后小规模复测一轮。

**预期效果**：明确 gstack 状态层不生效是环境因素还是 skill 自身设计（如遥测未授权即跳过），使"工具链增量价值"的结论有完整边界。
