# 报告：manual-superpowers 对比 empty 基线与根因分析（2026-06-11）

> 涉及运行：eval-manual-superpowers-260611093413（manual-skill 模式首跑）、eval-empty-260610231222（基线对照）
> 统一条件：DeepSeek deepseek-v4-pro · Claude Code 2.1.152 · 同一 spec 全量 M1~M8 · bypassPermissions
> 驱动方式差异：empty 为单条 prompt 自主跑完；manual-superpowers 按剧本逐轮 headless 注入 11 条 skill 流程提示（1× writing-plans → 8× executing-plans → verification-before-completion → requesting-code-review）

## 主题一：量化对比

| 指标 | empty（A） | manual-superpowers（B） | 倍率 |
|---|---|---|---|
| 历时 | 27.0 分钟 | 48.6 分钟 | 1.8× |
| 工具调用 | 107 | 316 | 3.0× |
| input tokens | 48.8 万 | 354 万 | 7.3× |
| output tokens | 28.1 万 | 35.8 万 | 1.3× |
| cache read | 1357 万 | 7495 万 | 5.5× |
| 代码规模 | 1869 行 / 15 文件 | 1855 行 / 16 文件 | ≈持平 |
| check-types / lint / 测试 | 全绿（33 例） | 全绿（29 例） | — |

B 多出的 ~209 次调用主要花在：每里程碑固定仪式（回读计划、四项全量检查、git commit，约 70~80 次）、2 次上下文压缩后的状态重建（Read 66 次 vs A 的 8 次）、更重的 @vscode/test-cli 基建、收尾 code review 及修复（~31 次）。

## 主题二：流程遵循观察

- **plan→execute→verify→review 主干真实遵循**：产出 3418 行计划文档（16 任务、含代码草稿与验证步骤）；逐里程碑执行并 git commit（A 全程零 git）；收尾派 code-review 子代理对照 spec 审查，**抓出 9 个问题含 4 个 CRITICAL**（侧栏按钮无响应、缩略图未转 webview URI、下载无超时、监听器泄漏）并修复。
- **TDD 名存实亡**：计划与实际执行均为"先实现后测试"，计划中 0 次出现 failing test，无红-绿循环。
- 修复循环错误率相近（A 9.3% vs B 7.6%）；B 修复次数多是因为检查频率高一个数量级，并非代码更差。
- 两边的"四检全绿"都不等于"能用"——差别在 B 有 review 工序去发现（呼应 260610-smoke-and-pipeline 主题二的结论）。

## 主题三：代码质量对比

- **按 spec 验收：A 明显更高**。A 的侧栏 UI（Radix 组件实际使用、卡片折叠、进度条、提交禁用态、错误列表、列数配置）与素材库链路（localResourceRoots 动态重设、插错文件警告）基本逐条落实；B 大面积缺失（Radix 装而不用、卡片不可折叠、`tasksCols` 收下改名 `_cols` 丢弃、协议死代码），且有伤害最大的缺陷：**`pollAll` 轮询全程不通知前端，进度条与缩略图在任务结束前不刷新**。
- **工程规范层：B 更好**。模块划分干净（10 个小模块 vs A 的 extension.ts 447 行大文件）、`ApiError` 类型安全错误模型（A 靠字符串匹配判定瞬时错误）、共享类型真正两端共用（A 在 webview 手抄一份类型，正是 spec 要避免的漂移）、mock fetch 真正测了 API 校验路径、用了 spec 指定的 @vscode/test-cli。
- 一句话：B 是"骨架更专业但只盖到一半的房子"，A 是"装修完整但接线偶有不规范的房子"。

## 主题四：根因分析——B 为何漏需求（不是爆上下文）

排除"爆 1M 上下文"假设：两次自动压缩都在 **~17 万 token** 触发；且侧栏 UI 主体（M2）完成于第一次压缩**之前**，压缩解释不了 UI 缺失。A 全程零压缩可作对照。

实际是**三层窗口收窄叠加**：

1. **第一层有损压缩（计划写作期）**：spec → 计划时 UI 细节就丢了。核对计划文档：错误列表 0 次提及、折叠仅 1 次、所有 UI 代码草稿中 Radix 组件使用 0 处、`pollAll` 草稿（计划 2090~2155 行）本身只有 `persist()` 没有 `emitUpdate()`——最严重缺陷是写计划时埋下的。
2. **第二层（执行期注意力锚定）**：spec 在整个实现期间只被读过一次（开局 01:34），直到收尾 review（02:16）才再读；中间 35 分钟只回读计划（17 次）。计划带完整代码草稿，执行退化成"誊写"——M2~M8 每里程碑仅 3~5 分钟。
3. **第三层（压缩后切片阅读）**：01:57 压缩把计划全文挤出上下文后，agent 再未完整读过计划，每轮只 Grep 定位 + Read 150~250 行当前任务切片；跨里程碑约束（如 M7 打磨进度条需知 M3 轮询要逐轮通知）不在切片内等于不存在。剧本每轮注入的是 skill 流程指令，从不重申 spec，进一步锁死注意力在"计划当前小节"这个最小窗口。

**结论**：失败链条 = 详尽 spec → 转述成计划（丢一层）→ 执行只忠于计划不回看 spec → 压缩后只剩切片。收尾 review 重读了 spec 故能抓回功能性 CRITICAL，但抽查式审查补不回计划阶段丢掉的长尾 UI 细节。当 spec 本身已足够详细时，"先计划后执行"的工作流等于做了一次不可靠的转述。

## 主题五：对应改进

### 改进一：监工模式——逐轮把 spec 送回上下文

**问题**：根因第 2、3 层表明，固定剧本每轮只重复 skill 流程指令，spec 在实现期 35 分钟内离场，压缩后连计划也只剩 250 行切片；剧本在 YAML 里预先写死，无法依据现场情况补救（计划本身也是第 1 轮才生成的，固定剧本不可能预先按轮切分计划）。

**方案**：`run-eval-manual-skill.sh` 拆出 `setup` / `round` / `export` 三个子命令（已落地并实测）：`setup` 只做环境准备并保留剧本文本与调用参数，轮次循环交由评测员接管。评测员每轮结束后读模型的轮次总结、核对实际改动、对照 spec 章节逐项验收，然后在剧本文本之外附**基于总结的大方向指导**——模拟真实用户的用法，只指出现象与需求出处（如"进度条在任务进行中不会动，对照 spec §8.4 自查"），不给代码级细节（不点名函数、不贴代码），缺陷由被测模型自己定位修复。正式评测统一改用此驱动方式，并为 empty 配平行剧本（`eval-config-manual-empty.yml`，逐轮与 superpowers 对应、仅不带 `/skill名` 前缀）作无 skill 对照组。监工协议全文记录于仓库根 CLAUDE.md。

**预期效果**：每轮验收把被测模型的注意力拉回 spec，长尾需求不再因计划转述与上下文压缩而静默丢失；同时指导停留在方向层，测出的仍是被测模型（及 skill）自身的修复与实现能力，而非评测员的。empty 对照组使 skill 净效果可以从监工增益中剥离。监工 empty 首跑已完成（结果见 `round3-01-260611-supervised-empty-trial.md`），superpowers 组待跑。

### 改进二：监工模式运行作为独立变体记录

**问题**：监工模式引入了"智能评测员"这一新变量，测的实际是"skill + 监工"的合成效果；若与 empty（全自主）、manual（固定剧本）的结果直接混比，结论会被驱动方式这个混淆变量污染。

**方案**：监工模式的 run 目录统一加 `supervised` 前缀（`eval-manual-supervised-<变体>-<时间戳>`，setup 子命令自动处理），报告中单列并注明驱动方式。

**预期效果**：三种驱动方式（全自主 / 固定剧本 / 监工）各自成列，跨组对比时驱动方式差异始终显式可见，避免误读为 skill 本身的优劣。

### 改进三：run 目录 git 隔离——堵住被测会话沿工作树越界

**问题**：监工模式首次试跑（eval-manual-supervised-empty-260611153916）中，被测模型第 2 轮把 M1 全部文件写进了评测仓库的 `image-flow/` 源目录（含改动仓库根 `.gitignore`），run 作废。根因：run 目录位于评测仓库 git 工作树内部且自己没有 `.git`，被测会话开场注入的 gitStatus 是评测仓库的（分支、历史提交、`image-flow/...` 相对路径俱全），模型据此把 `<仓库根>/image-flow` 认作项目根。取证时进一步发现同根因的既往污染：eval-manual-superpowers-260611093413 的逐里程碑 `git commit` 全部打进了评测仓库，当前的 `feature/m1-scaffolding` 分支即被测模型所建。另有剧透风险：剧本 `rounds/` 与调用参数 `.eval-env` 原先直接放在 run 目录内，被测模型可读到完整剧本。

**方案**：`run-eval-manual-skill.sh` 两处修改（已落地）。① setup 阶段在 run 目录 `git init` 独立仓库并提交初始材料基线，`.claude-home/` 写入 run 内 `.gitignore`——被测会话看到的 git 信息从此只有本目录；② 剧本参考与调用参数移入 `.claude-home/`（git 忽略且不在模型视线内），消除剧透。评测仓库内的污染（11 个文件 + `.gitignore` 改动）已清除还原。

**预期效果**：被测会话的文件读写与 git 操作天然圈定在 run 目录内，评测仓库不再被写入；隔离 git 仓库同时让"被测模型是否使用 git"恢复为可观测的行为指标（此前 superpowers 的提交越界落在评测仓库里，几乎被当成评测员自己的提交误读）。`feature/m1-scaffolding` 分支的处置（是否并回 main 或重建）留待人工决定。
