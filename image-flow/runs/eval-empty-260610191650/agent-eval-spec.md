# image-flow 从零开发需求文档（AI Agent 评测用）

> 本文档描述一个 VS Code 扩展的完整需求，用于评测 AI 编程 Agent：把本文档作为唯一输入，要求 Agent 从空目录开始实现全部功能。文档只规定**行为**，不规定具体实现写法。配套的验收评分标准在 `agent-eval-rubric.md`（评测者自留，不发给被测 Agent）。

## 1. 项目概述

开发一个名为 **image-flow** 的 VS Code 扩展：在编辑器内对 Markdown 文件调用 AI 绘图后端（Grsai API）生成图片。用户在 Markdown 里写提示词（正文即 prompt，可内嵌参考图），通过侧栏或右键菜单触发生成，图片自动下载到 Markdown 同级目录，并在侧栏中浏览任务进度、历史结果与素材库。

API 文档参见 `docs/grsai-api.md`（核心：`POST /v1/api/generate` 异步提交返回 job id，`GET /v1/api/result?id=` 轮询结果）。

## 2. 技术栈与构建要求

- TypeScript + esbuild，两条独立构建产物：
  - `dist/extension.js` — 扩展主进程（CommonJS，`vscode` 标记 external），是 `package.json` 的 `main` 入口。
  - `media/sidebar.js` — 侧栏 Webview 前端（React 19 + Radix UI，IIFE、platform=browser）。
- 样式为静态文件 `media/sidebar.css`，不经打包。
- 双 tsconfig 类型检查：主进程一套、webview（含 React/DOM 类型）一套；`npm run check-types` 跑两次 `tsc --noEmit`。
- ESLint flat config，`compile` / `package` 脚本前置执行 `check-types` 与 `lint`。
- 测试用 `@vscode/test-cli`（mocha），tsc 编译到 `out/` 供测试运行器使用。
- npm 脚本至少包含：`compile`、`watch`、`package`、`check-types`、`lint`、`test`。

## 3. 扩展清单（package.json contributes）

- 活动栏图标容器 `image-flow`，内含一个 webview 视图 `image-flow.sidebar`（名称「AI 绘图」）。
- 两条命令：
  - `image-flow.generateImage`（标题「Image Flow: 生成 AI 图片」）
  - `image-flow.previewRequest`（标题「Image Flow: 预览请求」）
- 资源管理器右键与编辑器右键菜单均挂这两条命令，仅当 `resourceExtname == .md` 时显示。
- `activationEvents: ["onStartupFinished"]` — VS Code 启动完成即激活，保证重启后能立即续拉未完成任务（见 §8）。
- 所有可释放对象（命令、监听器、任务管理器）必须 push 进 `context.subscriptions`。

## 4. 配置系统

### 4.1 配置项

| 字段 | 说明 | 默认值 |
|---|---|---|
| `apiKey` | Grsai API Key | 空 |
| `baseUrl` | 节点地址 | `https://grsai.dakka.com.cn`（国内节点）；可选 `https://grsaiapi.com`（全球节点） |
| `model` | 模型 | `nano-banana-2`；可选 `nano-banana-pro` / `gpt-image-2` / `gpt-image-2-vip` |
| `aspectRatio` | 比例 | `3:4`；可选 `1:1` / `16:9` / `9:16` / `4:3` |
| `imageSize` | 分辨率 | `1K`；可选 `2K` / `4K` |
| `concurrency` | 一次生成的并发张数 | 1（范围 1~10） |
| `workbenchCols` | 工作台素材缩略图每行张数 | 4（范围 1~8） |
| `tasksCols` | 任务页缩略图每行张数 | 2（范围 1~8） |
| `modelInjections` | 模型 → 注入提示词的映射 | 见 §4.3 |

### 4.2 存储要求

- `apiKey` 必须存 VS Code `secrets`（加密存储），**不得**进 `globalState`。
- 其余配置存 `globalState`，写入时按字段合并（patch 语义）。
- 素材库文件夹列表存 `workspaceState`（按工作区隔离）。

### 4.3 模型注入提示词

- 维护「模型 → 注入句」映射；生成时该句前置拼入最终 prompt（见 §7.3）。
- 内置种子：`gpt-image-2` 与 `gpt-image-2-vip` 预置「整体画面弱化微小细节，避免过度刻画。」；nano-banana 系列无种子。
- 首次激活时把种子写入配置，**只补配置中尚不存在的模型键**——用户改过（包括清空为空串）后不得再被种子覆盖。
- 激活时必须先完成种入再注册侧栏，保证侧栏首次读配置时默认句已可见。

## 5. Markdown 解析与参考图（图生图）

输入是一个 Markdown 文件，正文即提示词。需处理内嵌图片语法：

- 匹配 `![alt](路径)`，路径支持可选的 `<>` 包裹形式 `![alt](<带 空格(括号)的路径.png>)`——尖括号内可含空格与半角括号。
- 所有图片引用**按首次出现顺序去重编号**（同一路径复用同一序号）。
- 每张参考图相对 Markdown 所在目录读取，转为 `data:<mime>;base64,...` data URI，按编号顺序组成参考图数组随请求发送。
- 正文中每处图片语法替换为 `[imageN](文件名去扩展名)`，其余正文原样保留；替换编号必须与参考图数组顺序一致。
- 任何参考图读取失败：中断生成，报错并列出全部失败路径。
- Markdown 内容为空（trim 后）：报错「内容为空」，不发请求。

## 6. Grsai API 调用层

- `POST {baseUrl}/v1/api/generate`，`Authorization: Bearer <apiKey>`，请求体：`model`、`prompt`、`images`（参考图数组）、`replyType: 'async'` + 尺寸字段。
- **尺寸字段按模型系列区分**：
  - nano-banana 系列：传 `aspectRatio`（比例字符串）+ `imageSize`（1K/2K/4K）；
  - `gpt-image-2`：只传 `aspectRatio` 比例，不带 `imageSize`；
  - `gpt-image-2-vip`：只认像素值，需按「比例 + 分辨率」查表换算成 `宽x高` 填进 `aspectRatio`（如 3:4 + 1K → `864x1152`；全表见 API 文档）。
- 提交成功返回 job id；返回 `failed`/`violation` 或无 id 视为提交失败。
- `GET {baseUrl}/v1/api/result?id=` 查询：`running`（可带 progress 0~100）/ `succeeded`（带 results 图片 url 列表）/ `failed` / `violation`（带 error）。
- **响应必须做运行时校验**（status 枚举、id/results/error 形状），不得裸断言。
- 所有网络请求带超时（30s 量级），超时 abort。
- 查询接口返回 HTTP 5xx / 429 视为**可重试的瞬时错误**（保持任务 running，下轮重试）；网络层失败（fetch failed / abort）同样按瞬时处理；响应格式异常、写盘失败等则直接判失败，不无限重试。

## 7. 生成流程（核心异步任务机制）

### 7.1 提交

点击生成（侧栏按钮或右键命令）后：

1. 校验 apiKey 已配置，否则提示去设置页填写。
2. 读取 Markdown → 解析参考图 → 组装最终 prompt（§7.3）。
3. 在 Markdown 同级创建任务文件夹 `task-<yyMMddHHmmSS>-<seq>`（时间戳精确到秒，同秒连续提交靠递增 seq 区分，不得互相覆盖）。
4. **立即建卡返回**：本地任务卡先入列（N 个 job 全部置 `submitting` 态），UI 不等网络往返；随后后台并发发出 N 个 generate 请求（N = 并发数配置），逐个拿到 job id 转 `running`。
5. 全部 N 个提交都失败：整任务作废、从列表移除、删除空文件夹、弹错误通知（聚合去重后的错误信息）。
6. 提交成功后侧栏自动切到「任务」标签页。

### 7.2 轮询与下载

- 所有进行中任务共用**一个**定时器（4s 间隔），仅当存在 running job 时运转，无则停掉。
- 轮询必须有重入锁：一轮未结束（下载图片可能超过间隔）不得开启下一轮。
- 某 job `succeeded`：把结果 url 列表下载到任务文件夹，文件名 `{md文件名}-{序号}.{扩展名}`，序号在整个任务内**接续编号**（多 job 不得重名互相覆盖）；扩展名取自 url 并校验在图片白名单内，否则回退 png。
- 某 job `failed`/`violation`：记录错误。
- 任务超时兜底（10 分钟）：仍 running 的 job 标记失败，防止僵死记录永不清除。
- 任务全部 job 终结后从持久化列表移除；若有失败 job，按「有成图→警告通知 / 无成图→错误通知」弹窗（错误去重聚合）；无任何成图的任务移除前删除其空文件夹。

### 7.3 提示词注入

最终 prompt = `模型注入句` + `工作区根目录 IMAGES.md 全文` + `替换后正文`，按此固定顺序用空行连接，空段省略。IMAGES.md 不存在 / 为空 / 无工作区均静默跳过，不阻断生成。提交与预览（§9）必须走同一拼装逻辑。

### 7.4 持久化与重启续拉

- 进行中任务（含 job id、状态、已下载图片、时间戳）持久化进 `globalState`，状态变更随时落盘。
- 扩展激活时 `resume()`：若有未完成任务，立即拉一次并恢复定时轮询。
- 超时计时按「本次会话起算」重置——关机时长不计入，离线超 10 分钟的任务不得被误判超时。
- 任务真实开始时间（用于 UI 显示已进行时长）单独保存，重启**不**重置。
- 提交途中被关闭的 job（`submitting` 态无 id）重启后直接判失败（「提交未完成」），不得永久卡住。

## 8. 侧栏 UI（Webview + React）

侧栏为单页 React 应用，三个标签页：**工作台 / 任务 / 设置**。切换标签保留各页内部状态（展开态等）。

### 8.1 通用机制

- 前后端用 `postMessage` 双向通信，消息协议与共享类型集中在单一模块（如 `src/shared.ts`），两端共用，避免漂移。
- 前端挂载后发 `init`，扩展回推配置、当前 MD、进行中任务、历史、素材库。
- Webview HTML 必须带 CSP：脚本锁 nonce；图片与样式限 `webview.cspSource`（样式可放开 unsafe-inline 以兼容 Radix 内联定位）。
- 本地图片必须经 `asWebviewUri` 转换后才能在 webview 显示；`localResourceRoots` 包含扩展 media 目录、工作区目录与全部素材库目录，素材库增删后需重设。
- 配置项**单字段即时保存**，无保存按钮。

### 8.2 生效 Markdown 跟随

- 侧栏关联一个「生效 MD」：跟随当前活动编辑器，切到某 `.md` 即生效；切到非 Markdown 标签（预览、图片详情等）时**保留上一个生效 MD 不清空**。
- 生效 MD 变化时刷新该 MD 的历史与自动素材库；未变化不重扫。
- 右键菜单触发生成时，目标 MD 同时成为生效 MD 并聚焦侧栏。

### 8.3 工作台页

- 上部：素材库区（§10）。
- 底部停靠区：
  - 一行四个控件：模型 / 分辨率 / 比例 下拉 + 并发数步进器（1~10）。
  - 生成行：显示当前生效 MD 文件名（无则「未打开 Markdown 文件」）+「预览请求」按钮 +「生成」按钮（提交期间禁用并显示「生成中…」）。
  - 状态行：显示最近一次状态/错误消息（错误红色）。

### 8.4 任务页

- 进行中任务与历史任务**合并为一条按文件夹名（含时间戳）倒序的列表**，新任务在前。
- 进行中卡片（默认展开，可折叠）：标题 = `文件夹名（模型） · 提交中 x/N 或 生成中 done/N · [失败 n ·] 已进行时长`；展开后显示——
  - 聚合进度条 + 百分比：每个 job 占 1/N，终结 job 记满分、running 取远端进度、submitting 记 0，取整均摊；
  - 已下载图片的缩略图（随 job 完成实时增加）；
  - 「正在提交 n 个请求…」「还有 n 张正在生成…」提示与错误列表。
  - 已进行时长每秒自增，格式 `mm:ss`（超 1 小时 `h:mm:ss`），基于真实提交时间。
- 历史卡片（默认收起）：标题 = `文件夹名（n 张）`，展开显示缩略图。
- 历史来源：扫描生效 MD 同级目录的全部 `task-*` 文件夹内图片，倒序；**排除进行中任务的文件夹**避免与进行中卡片重复；无图片的文件夹不列出。
- 缩略图点击用 VS Code 打开原图。每行张数由 `tasksCols` 控制。
- 任务完成的瞬间不得闪烁：先把完成图并入历史，再移除进行中卡片。

### 8.5 设置页

- API Key 密码输入框 + 「获取 API Key →」链接（`https://grsai.ai/zh/dashboard/api-keys`，经扩展用系统浏览器打开）。
- 节点下拉（国内/全球）。
- 工作台每行张数、任务栏每行张数两个步进器。
- 模型注入提示词编辑：一个模型下拉（默认当前工作台模型，独立选择不影响工作台）+ 对应模型注入句的多行文本框。

## 9. 预览请求（不消耗额度）

「预览请求」命令/按钮：对目标 MD 走与真实提交**完全一致**的解析与请求体构造，但不调 API，把结果打开成一个临时预览文档，包含四段：

1. 最终提示词（含注入，单独成段，不转义）；
2. 请求地址（`POST {baseUrl}/v1/api/generate`）;
3. 请求参数 JSON（剔除 prompt 与 images 字段，避免长 base64 刷屏）；
4. 参考图概览（每张列出 `imageN: data URI 前 48 字符…（总长度）`，无参考图则注明）。

## 10. 素材库

工作台素材库区分两段，缩略图行为一致：左键打开原图，**右键把图片以相对路径 Markdown 引用插入生效 MD 的光标处**。

### 10.1 自动素材库（「当前路径」）

- 按生效 MD 的路径自动生成：从工作区根的下一层目录起，到 MD 所在目录为止，**每一层各成一个库**（不含工作区根那层），只取该层目录里直接的图片（不递归）。
- 无图片的层不显示；MD 不在工作区内则为空。

### 10.2 手动素材库

- 「+ 添加」弹系统文件夹选择器，选中目录加入（去重）；每个库可单独移除。
- 每个库**递归**扫描图片，深度上限 3 层、条目数上限 500（防止误加超大目录卡死）。
- 列表存 `workspaceState`。

### 10.3 插入引用规则

- 仅当当前活动编辑器**正是**生效 MD 时才插入，否则警告并忽略（防插错文件）。
- 路径取相对 MD 目录的 POSIX 风格相对路径，非 `..` 开头时加 `./` 前缀。
- 路径含空格或半角括号时用 `<>` 包裹（与 §5 的解析端对应）；跨盘符无法相对引用时警告并放弃。
- alt 文本取图片文件名（去扩展名）。

## 11. 图片类型白名单

统一维护图片扩展名白名单（至少 png/jpg/jpeg/gif/webp/bmp/svg 量级），供素材库扫描、历史扫描、下载扩展名校验与 mime 推断共用。

## 12. 测试要求

至少为以下纯逻辑写单元测试（vscode-test + mocha）：

- 图片语法解析与替换：普通路径、尖括号路径（含空格/括号）、重复引用复用编号、编号与顺序一致；
- 聚合进度计算（全 submitting=0、混合态、全部终结=100）；
- 瞬时错误判定（5xx/429/abort/fetch failed 为瞬时；格式错误非瞬时）；
- gpt-image-2-vip 像素换算；
- 预览文本组装（剔除 prompt/images、参考图概览格式）。

## 13. 建议实现顺序（分阶段里程碑）

按以下顺序由易到难推进，每个阶段产出都可独立运行验证：

1. **M1 脚手架 + 端到端生成**：扩展骨架、右键 Markdown 触发生成、参考图解析（图生图）、多模型尺寸字段区分。此阶段可先用同步等待实现，跑通端到端。
2. **M2 侧栏化与配置系统**：Webview 侧栏 + React 前端、配置自管（globalState + secrets）、三标签页、即时保存。
3. **M3 异步任务机制**：改为 `replyType:async` 提交 + 单定时器后台轮询 + globalState 持久化 + 重启续拉；多任务并发互不干扰。
4. **M4 素材库**：手动素材库（递归扫描、上限保护）与按 MD 路径逐层自动生成的「当前路径」库；右键缩略图插入相对引用（含空格路径 `<>` 包裹及解析端兼容）。
5. **M5 预览请求**：与真实提交共用请求体构造的预览文档，不调 API。
6. **M6 提示词注入**：`modelInjections` 配置 + 首次激活种入种子（不覆盖用户改动）+ IMAGES.md 注入，提交与预览两处生效。
7. **M7 体验打磨**：后台提交（点生成立即建卡不等网络）、进行中卡片进度条与逐秒计时、生效 MD 跟随活动编辑器（非 MD 标签不清空）、任务/历史合并倒序列表。
8. **M8 健壮性收尾**：响应运行时校验、瞬时错误重试判定、空文件夹清理、轮询重入锁、纯函数抽取与单元测试补齐。
